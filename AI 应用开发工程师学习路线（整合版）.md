# AI 应用开发工程师学习路线（整合优化版）

> **核心定位**：将大模型作为高性能 API 集成到业务系统中，而非从头训练模型
> 
> **学习周期**：全职 3-5 个月 | 业余 6-8 个月（每天 1.5-2 小时）
> 
> **核心原则**：不啃高数、不读论文、不预训练模型、重实操轻理论

---

## 零、核心认知（转行前必读）

### 岗位区分
| 岗位类型 | 工作内容 | 技能要求 | 适合人群 |
|---------|---------|---------|---------|
| **AI 应用开发** | 调用模型、RAG 开发、Agent 编排、业务集成 | Python/Java、框架使用、工程化 | **零基础转行首选** |
| 算法工程师 | 训模型、调参、优化 | 高数、线代、深度学习 | 需科班背景 |

```
     ↓           ↓            ↓              ↓       ↓         ↓
  编程能力    接口封装      框架使用      知识库    工作流    Docker/监控
```

## 第一阶段：基础筑基（2-4 周）

### 1.1 Python 工程能力（2-3 周）

**核心目标**：能独立编写可维护的 API 服务，处理数据流

| 模块     | 必学内容                               | 避坑指南                 |
| ------ | ---------------------------------- | -------------------- |
| 基础语法   | 变量/循环/函数/异常处理/JSON 解析              | 不用学爬虫/数据分析/可视化       |
| 数据处理   | `pandas`清洗数据、`numpy` 基础（会调 API 即可） | 无需手推公式               |
| Web 框架 | **FastAPI**（必学）实现 RESTful 接口       | 优先实战，避免纯理论           |
| 工程规范   | `Pydantic` 校验、`Git` 版本控制、`uv` 包管理  | 安装时指定`--python=3.10` |

**实战任务**：用 FastAPI 写一个天气查询接口（调用第三方 API）

### 1.2 大模型基础认知（1-2 周）

**核心目标**：理解大模型能做什么、不能做什么

| 关键概念 | 说明 |
|---------|------|
| Token 机制 | 1 个汉字≈2 个 Token，成本直接关联消耗量 |
| 能力边界 | 擅长文本生成/归纳，不擅长精确计算/实时数据 |
| API 调用 | 掌握 `temperature`、`max_tokens` 参数调优 |
| 主流模型 | 通义千问、DeepSeek、文心一言、讯飞星火、Claude |

**实战任务**：
- 注册 2 个大模型平台，对比同一 Prompt 的输出差异与延迟
- 用 Postman 直接测试 API，观察输入输出结构

## 第二阶段：核心技术专项（8-12 周）

### 2.1 Prompt 工程（1 周）

**核心目标**：让大模型稳定输出符合业务需求的内容

| 技巧 | 说明 | 应用场景 |
|-----|------|---------|
| 标准结构 | 角色 + 背景 + 任务 + 约束 + 输出格式 | 所有场景 |
| Few-Shot | 提供示例引导输出 | 复杂任务 |
| CoT 思维链 | 让模型分步推理 | 逻辑计算 |
| 结构化输出 | 强制 JSON/XML 格式 | 数据提取 |

**实战任务**：编写 3 个高质量 Prompt，实现稳定输出指定格式的 JSON

### 2.2 RAG 检索增强生成（4-6 周）⭐核心重点

**核心目标**：解决大模型"幻觉"问题，构建基于私有数据的精准问答系统

#### RAG 完整流程
```
文档上传 → 文本切片 → Embedding 向量化 → 向量入库 → 问题向量化 → 相似检索 → 生成答案
```

#### 技术栈选型
| 环节 | 工具选择 | 说明 |
|-----|---------|------|
| 文档解析 | `Unstructured`/`PyPDF2`/`Docling` | PDF/Word/Excel 解析 |
| 文本分块 | `RecursiveCharacterTextSplitter` | 按段落/语义切片 |
| 向量化 | `OpenAIEmbeddings`/`bge-m3`/`Qwen-Embedding` | 文本转向量 |
| 向量库 | `Chroma`(轻量) / `Milvus`/`Qdrant`(生产) | 存储检索 |
| 开发框架 | `LangChain` / `LlamaIndex`（二选一先精） | 链式编排 |
| 检索优化 | 混合检索 ( 关键词 + 向量) + `CohereRerank` 重排序 | 提升准确率 |

#### 关键优化点
- **数据质量决定 80% 效果**：优先做好数据清洗、重叠窗口设计
- **切片策略**：对比不同 chunk_size 对检索准确率的影响
- **兜底策略**：检索不到答案时的降级方案

**实战任务**：
- 入门：用 LangChain 搭建"公司制度问答机器人"，上传 PDF 后精准回答
- 进阶：企业 RAG 知识库系统（Milvus+ 文档解析 + 权限控制）

### 2.3 Agent 智能体开发（3-4 周）⭐高薪加分

**核心目标**：让 AI 从"被动问答"升级为"主动执行任务"

#### 核心架构
| 模式 | 说明 | 框架 |
|-----|------|------|
| ReAct | 推理 + 行动交替 | LangChain |
| Plan-and-Execute | 先规划后执行 | LangGraph |
| Function Calling | 工具调用机制 | 原生支持 |

#### 技术栈
- **框架**：`LangChain`（生态全）/ `LangGraph`（复杂流程）/ `AutoGen`（多 Agent）
- **低代码**：`Dify` / `Coze`（快速原型）
- **记忆机制**：框架内置 + `mem0`/`zep` 补充
- **上下文工程**：长短期记忆管理

**实战任务**：
- 开发"能查天气/写邮件/调数据库"的多工具 Agent
- 搭建客服工单自动分派 + 知识库辅助回复的工作流

### 2.4 本地模型部署（1 周）

| 工具 | 说明 | 适用场景 |
|-----|------|---------|
| **Ollama** | 一键部署开源模型，CPU 可跑 | 本地开发/离线环境 |
| vLLM | 推理加速 | 高并发生产 |
| 模型选型 | Qwen2.5/Llama3/GLM | 根据场景选择 |

**实战任务**：本地跑通 7B 模型，替换云端 API，对比成本与延迟

## 第三阶段：工程化与部署（3-4 周）

### 3.1 后端服务封装

| 能力 | 技术实现 | 说明 |
|-----|---------|------|
| RESTful API | `FastAPI` + `Uvicorn` | 标准接口设计 |
| 流式响应 | SSE（Server-Sent Events） | 打字机效果 |
| 异步处理 | `async/await` + 消息队列 | 长耗时任务 |
| 并发控制 | `Redis` 缓存 + 限流 | 防止过载 |

### 3.2 LLMOps 基础

- **监控**：`LangSmith`/`Phoenix` 追踪 Token 消耗、延迟、错误率
- **评估**：`Ragas` 自动化测试回答准确性、相关性
- **成本控制**：缓存策略 + 模型路由（简单任务用小模型）
- **Prompt 版本管理**：迭代优化记录

### 3.3 容器化部署

```bash
# 核心能力
Docker 环境隔离 → Dockerfile 依赖打包 → docker-compose 一键部署 → 云服务器上线
```

### 3.4 企业级能力

| 能力 | 实现方式 |
|-----|---------|
| 数据安全 | 敏感词过滤、内容脱敏 |
| 身份鉴权 | OAuth2、API Key |
| 日志埋点 | 调用量统计、计费统计 |
| 合规风控 | 回答置信度评分、来源标注 |

## 第四阶段：项目实战（贯穿全程）

### 项目组合（必做 3 个）

#### 项目 1：RAG 知识库问答系统（核心项目）
```
技术栈：FastAPI + LangChain + Milvus + 文档解析
功能：
- 上传 PDF/Word 自动解析、分段存储
- 文档向量化入库
- 私有知识库问答、精准资料引用
- 文档管理、权限控制
指标：文档解析准确率>90%，幻觉率下降 70%
```

#### 项目 2：多工具 Agent 工作流
```
技术栈：LangGraph + FastAPI + 第三方 API
功能：
- 工具调用（搜索/数据库/邮件）
- 多步骤流程编排
- 人工介入（Human-in-the-loop）
- 异常重试机制
指标：任务完成率>85%，死循环率<1%
```

#### 项目 3：AI 对话中台（工程化项目）
```
技术栈：FastAPI + Redis + MySQL + SSE
功能：
- 多模型管理（通义/DeepSeek 切换）
- 多轮会话保存、历史对话查询
- 流式问答输出
- Token 统计、费用管控、限流熔断
指标：QPS 支持 50+，单次成本<0.1 元
```

### 作品集要求
- 代码开源至 GitHub，含清晰 README
- 架构图 + 部署步骤 + 演示视频
- 成本分析报告（优化前后对比）
