基于 Andrej Karpathy 公开的 LLM Wiki 理念与多篇实战资料，我为你整理了一套**可直接交付给 Claude Code / OpenCode 等 AI 编程 Agent 执行的完整搭建方案**。方案严格遵循“知识编译”范式，强调职责分离、持久化沉淀与自动化维护。

---

### 📦 一、标准目录结构（先在本地创建）

在你的电脑中新建文件夹（如 `my-llm-wiki`），并按以下结构初始化：

```text
my-llm-wiki/
├── raw/                # 📥 原始素材库（人工只放，AI只读）
│   ├── _inbox/         # 待处理暂存区（新资料先丢这里）
│   └── ...             # AI 后续自动创建的子目录（articles/papers/notes等）
├── wiki/               # 📘 编译产物库（AI 专属维护，人工只读）
│   ├── summaries/      # 单篇资料结构化摘要
│   ├── concepts/       # 跨文献提炼的概念/术语页
│   ├── entities/       # 人物/公司/工具/框架页
│   └── analysis/       # 问答沉淀/对比分析/深度综合
├── logs/               # 📜 系统日志与索引
│   ├── index.md        # Wiki 页面总索引（标题 | 路径 | 一句话简介）
│   └── log.md          # 变更日志（按时间倒序）
└── AGENTS.md           # 🤖 AI 行为规范（核心提示词，见下文）
```

> ✅ 创建后，用 Obsidian 的 `Open folder as vault` 打开 `my-llm-wiki` 即可预览。

### 🤖 二、核心提示词（直接保存为 `AGENTS.md`）

将以下内容完整复制，保存为项目根目录的 `AGENTS.md`。Claude Code / OpenCode 读取此文件后，即可自动进入“Wiki 编译员”状态。

```markdown
# 🤖 LLM Wiki 编译员行为规范 (AGENTS.md)

## 🎯 核心定位
你是我的“知识编译器”。你的任务是将 `raw/` 中的原始资料，持续编译为 `wiki/` 中结构化、可交叉引用、自动演进的 Markdown 知识库。你需替代人工完成摘要、分类、打链、冲突标注与索引维护。

## 📁 目录权限与职责
| 目录 | 写入方 | 读取方 | 规则 ||---|---|---|---||`raw/`| 仅人类 | 人类+AI |**绝对只读**。禁止移动、重命名、修改或删除任何文件。 ||`wiki/`| 仅AI | 人类+AI | AI 全权维护。页面聚焦单一主题，大量使用 `[[内部链接]]`。 ||`logs/`| 仅AI | 人类+AI |`index.md` 维护全局目录；`log.md` 记录每次操作变更。 |
## ⚙️ 工作流 1：资料摄入与编译 (Ingestion)
当 `raw/` 中出现新文件时，严格按序执行：
1. **读取解析**：完整读取文件，提取核心观点、关键数据、方法论、实体名称。
2. **编译产出**：
   - 单篇资料 → `wiki/summaries/[安全文件名].md`
   - 新概念/理论 → `wiki/concepts/[概念名].md`
   - 人物/工具/框架 → `wiki/entities/[名称].md`
3. **交叉链接**：
   - 在新页面中用 `[[已有页面]]` 关联相关知识。
   - 在被链接页面的末尾追加 `🔗 相关引用：[[本页面]]`。
4. **冲突处理**：若新内容与已有结论矛盾，**严禁静默覆盖**。必须在对应页面标注：`⚠️ 知识冲突：[旧观点] vs [新证据] | 来源：xxx`。
5. **更新日志**：
   - 同步刷新 `logs/index.md`（格式：`- [[页面标题]] | 路径 | 一句话简介`）
   - 在 `logs/log.md` 顶部追加记录：`- [YYYY-MM-DD HH:mm] 处理: xxx.md | 新增/更新: xxx | 关键变更: xxx`

## 💬 工作流 2：问答与知识沉淀 (Query)
当用户提问时：
1. 优先读取 `logs/index.md` 定位相关 wiki 页面。
2. 深度阅读对应 `wiki/` 页面，综合提炼答案。
3. 若回答具备长期复用价值，将其结构化保存为 `wiki/analysis/[主题问答].md`，并更新索引与日志。
4. 输出答案时，必须附带引用的 `[[Wiki 页面]]` 保证可追溯性。

## 🚫 强制约束
- 仅使用标准 Markdown，禁止输出 HTML/复杂表格/占位符。
- 每次操作后自检：内部链接是否有效？`index.md` 是否同步？`log.md` 是否记录？
- 遇到无法解析的文件（加密PDF/乱码/二进制），记录至 `logs/log.md` 并跳过。
- 保持“单一事实来源”，绝不重复创建语义相同的页面。
```

### 🚀 三、在 Claude Code / OpenCode 中的执行步骤

#### 1️⃣ 初始化 Agent 上下文

打开终端（或 IDE 内置终端），进入 `my-llm-wiki` 目录：

```bash
cd my-llm-wiki
# Claude Code 示例：
claude
# 进入对话后输入：
@AGENTS.md 请完整阅读此文件，理解 LLM Wiki 编译理念。确认后回复“✅ Wiki 编译员已就绪，请放入 raw/_inbox 资料或下达指令。”
```

#### 2️⃣ 首次批量编译测试

将 2~3 篇 PDF/网页/笔记丢入 `raw/_inbox/`，然后在终端输入：

```text
请扫描 raw/_inbox/ 中的新资料，执行 Ingestion 工作流。完成后汇报处理清单与日志变更。
```

#### 3️⃣ 日常提问与沉淀

```text
基于当前 wiki 库，解释“检索增强生成(RAG)与 LLM Wiki 编译模式的核心差异”，并附上引用的 wiki 页面链接。若结论具有长期价值，请沉淀至 wiki/analysis/。
```

### 💡 四、关键避坑与进阶建议

|问题|解决方案|
|---|---|
|**AI 幻觉/乱建链接**|在提示词中已强制要求“先读 index.md 再操作”，Obsidian 开启 `Link Checker` 插件可一键标红断链。|
|**上下文窗口溢出**|Karpathy 原话强调：每次对话聚焦单次 Ingestion 或单次 Query。超过 5 篇长文档请分批喂入 `raw/_inbox/`。|
|**旧笔记迁移**|将历史笔记整体移至 `legacy/` 目录（AI 设为只读）。需要时让 AI 逐步“重编译”至新 `wiki/` 结构，避免污染。|
|**自动化增强**|可搭配 Obsidian 插件 `Auto Note Mover`（自动归位 raw 文件）或 `Templater`（生成标准化摘要模板），进一步降低人工干预。|

---

### 📌 总结：为什么这套方案优于传统 RAG？

- **RAG**：提问时临时检索 → 每次从零拼凑 → 无知识沉淀 → 答完即忘。
- **LLM Wiki**：存入时即时编译 → 持续更新交叉引用 → 矛盾显式标注 → 知识产生复利。

你只需做两件事：`📥 丢资料进 raw/` 和 `💬 提问`。剩下的摘要、分类、打链、维护，全部交由 AI Agent 按 `AGENTS.md` 自动执行。打开 Obsidian 的图谱视图，你会看到知识网络随着每次编译自动生长。
