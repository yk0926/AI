# LLM Wiki

一种使用 LLM 构建个人知识库的模式。

这是一个想法文件，设计用于复制粘贴到你自己的 LLM 代理（例如 OpenAI Codex、Claude Code、OpenCode / Pi 等）。它的目标是传达高层次的想法，但你的代理会与你协作构建具体细节。

## 核心思想

大多数人对 LLM 和文档的体验都是 RAG 模式：你上传一批文件，LLM 在查询时检索相关片段，然后生成答案。这能工作，但 LLM 每次都在从零开始重新发现知识。没有积累。问一个需要综合五份文档的微妙问题，LLM 每次都不得不找到并拼凑相关片段。没有任何东西被构建起来。NotebookLM、ChatGPT 文件上传和大多数 RAG 系统都是这样工作的。

这里的想法不同。LLM 不是在查询时从原始文档检索，而是**增量构建并维护一个持久化的 wiki**——一个结构化的、相互链接的 markdown 文件集合，位于你和原始来源之间。当你添加新来源时，LLM 不只是为后续检索建立索引。它会阅读、提取关键信息，并将其整合到现有 wiki 中——更新实体页面、修订主题摘要、指出新数据与旧主张相矛盾的地方、加强或挑战不断演变的综合。知识被编译一次，然后*保持最新*，而不是每次查询都重新推导。

这是关键区别：**wiki 是一个持久的、不断累积的产物。** 交叉引用已经在那里。矛盾已经被标记。综合已经反映了你读过的所有内容。每添加一个来源、每问一个问题，wiki 就变得更丰富。

你从不（或很少）自己写 wiki——LLM 编写并维护所有内容。你负责来源、探索和提出正确的问题。LLM 做所有繁琐的工作——总结、交叉引用、归档和记录，这些工作使知识库真正长期有用。在实践中，我让 LLM 代理打开在一侧，Obsidian 打开在另一侧。LLM 根据我们的对话进行编辑，我实时浏览结果——跟踪链接、查看图形视图、阅读更新的页面。Obsidian 是 IDE；LLM 是程序员；wiki 是代码库。

这可以应用于许多不同的场景。一些例子：

- **个人**：追踪自己的目标、健康、心理学、自我提升——归档日记条目、文章、播客笔记，并随着时间的推移构建自己的结构化图景。
- **研究**：深入一个主题数周或数月——阅读论文、文章、报告，并逐步构建一个具有不断演变的论点的综合 wiki。
- **阅读一本书**：边读边归档每一章，为角色、主题、情节线索建立页面，以及它们如何连接。到最后你拥有一个丰富的伴侣 wiki。想象一下像 [Tolkien Gateway](https://tolkiengateway.net/wiki/Main_Page) 这样的粉丝 wiki——成千上万相互链接的页面涵盖角色、地点、事件、语言，由志愿者社区多年构建。你可以在阅读时个人构建类似的东西，让 LLM 完成所有交叉引用和维护工作。
- **商业/团队**：由 LLM 维护的内部 wiki，由 Slack 线程、会议记录、项目文档、客户电话提供内容。可能有人类在循环中审查更新。wiki 保持最新，因为 LLM 做了团队中没人想做的维护工作。
- **竞争分析、尽职调查、旅行计划、课程笔记、爱好深入**——任何你随着时间的推移积累知识并希望它被组织而不是分散的场景。

## 架构

有三层：

**原始来源**——你策划的来源文档集合。文章、论文、图像、数据文件。这些是不可变的——LLM 从它们读取但从不修改它们。这是你的真实来源。

**wiki**——LLM 生成的 markdown 文件目录。摘要、实体页面、概念页面、比较、概述、综合。LLM 完全拥有这一层。它创建页面、在新来源到达时更新它们、维护交叉引用，并保持一切一致。你阅读它；LLM 编写它。

**schema**——一个文档（例如 Claude Code 的 CLAUDE.md 或 Codex 的 AGENTS.md），告诉 LLM wiki 如何结构化、约定是什么，以及在摄取来源、回答问题或维护 wiki 时要遵循的工作流程。这是关键的配置文件——它使 LLM 成为一个有纪律的 wiki 维护者，而不是通用的聊天机器人。你和 LLM 随着时间共同演进它，因为你弄清楚了什么对你的领域有效。

## 操作

**摄取。** 你将新来源放入原始集合并告诉 LLM 处理它。一个示例流程：LLM 阅读来源，与你讨论关键要点，在 wiki 中编写摘要页面，更新索引，更新整个 wiki 中相关的实体和概念页面，并将条目附加到日志中。单个来源可能触及 10-15 个 wiki 页面。就我个人而言，我更喜欢一次摄取一个来源并保持参与——我阅读摘要、检查更新，并指导 LLM 强调什么。但你也可以一次性批量摄取多个来源，减少监督。由你开发适合你风格的工作流程，并将其记录在 schema 中供未来会话使用。

**查询。** 你针对 wiki 提出问题。LLM 搜索相关页面、阅读它们，并综合一个带引用的答案。答案可以采取不同形式，取决于问题——markdown 页面、比较表、幻灯片（Marp）、图表（matplotlib）、画布。重要的见解：**好的答案可以作为新页面归档回 wiki。** 你要求的比较、分析、你发现的连接——这些是有价值的，不应该消失在聊天记录中。这样你的探索就像摄取的来源一样在知识库中累积。

**清理。** 定期让 LLM 对 wiki 进行健康检查。寻找：页面之间的矛盾、被新来源取代的过时主张、没有入站链接的孤立页面、重要概念被提及但缺乏自己的页面、缺失的交叉引用、可以通过网络搜索填补的数据空白。LLM 擅长建议新的调查方向和新的来源寻找。这使 wiki 在增长时保持健康。

## 索引和日志

两个特殊文件帮助 LLM（和你）在 wiki 增长时导航它们。它们服务于不同的目的：

**index.md** 是面向内容的。它是 wiki 中所有内容的目录——每个页面都列出链接、一行摘要，以及可选的元数据如日期或来源计数。按类别组织（实体、概念、来源等）。LLM 在每次摄取时更新它。回答问题时，LLM 首先阅读索引以找到相关页面，然后深入它们。这在中等规模下效果出奇的好（约 100 个来源，数百个页面），避免了基于嵌入的 RAG 基础设施的需求。

**log.md** 是按时间顺序的。它是一个仅追加的记录，记录发生了什么以及何时发生——摄取、查询、清理。一个有用的提示：如果每个条目都以一致的前缀开头（例如 `## [2026-04-02] ingest | Article Title`），日志就可以用简单的 unix 工具解析——`grep "^## \[" log.md | tail -5` 给你最后 5 个条目。日志为你提供 wiki 演变的时间线，帮助 LLM 了解最近完成了什么。

## 可选：CLI 工具

在某些时候，你可能想要构建小工具来帮助 LLM 更有效地操作 wiki。最明显的是 wiki 页面上的搜索引擎——在小规模下索引文件就足够了，但随着 wiki 增长，你需要适当的搜索。[qmd](https://github.com/tobi/qmd) 是一个不错的选择：它是一个用于 markdown 文件的本地搜索引擎，具有混合 BM25/向量搜索和 LLM 重新排序，全部在设备上运行。它有 CLI（因此 LLM 可以调用它）和 MCP 服务器（因此 LLM 可以将其用作原生工具）。你也可以自己构建更简单的东西——LLM 可以帮助你在需要时快速编写一个简单的搜索脚本。

## 技巧和窍门

- **Obsidian Web Clipper** 是一个浏览器扩展，可将网页文章转换为 markdown。对于快速将来源放入原始集合非常有用。
- **在本地下载图像。** 在 Obsidian 设置 → 文件和链接中，将"附件文件夹路径"设置为固定目录（例如 `raw/assets/`）。然后在设置 → 热键中，搜索"Download"找到"Download attachments for current file"并将其绑定到热键（例如 Ctrl+Shift+D）。剪辑文章后，按热键，所有图像都会下载到本地磁盘。这是可选的但有用——它让 LLM 直接查看和引用图像，而不是依赖可能失效的 URL。注意 LLM 无法一次性阅读带有内联图像的 markdown——解决方法是让 LLM 先阅读文本，然后单独查看一些或所有引用的图像以获得更多上下文。有点笨拙但足够好用。
- **Obsidian 的图形视图** 是查看 wiki 形状的最佳方式——什么连接到什么、哪些页面是枢纽、哪些是孤立的。
- **Marp** 是一种基于 markdown 的幻灯片格式。Obsidian 有一个插件。用于直接从 wiki 内容生成演示文稿。
- **Dataview** 是一个 Obsidian 插件，可在页面 frontmatter 上运行查询。如果你的 LLM 向 wiki 页面添加 YAML frontmatter（标签、日期、来源计数），Dataview 可以生成动态表格和列表。
- wiki 只是一个 markdown 文件的 git 仓库。你免费获得版本历史、分支和协作功能。

## 为什么这有效

维护知识库的繁琐部分不是阅读或思考——而是记录。更新交叉引用、保持摘要最新、注意新数据何时与旧主张矛盾、在数十个页面中保持一致性。人类放弃 wiki 是因为维护负担增长快于价值。LLM 不会感到无聊，不会忘记更新交叉引用，可以在一次通过中触及 15 个文件。wiki 保持维护状态，因为维护成本接近于零。

人类的工作是策划来源、指导分析、提出好问题，并思考这一切意味着什么。LLM 的工作是其他一切。

这个想法在精神上与 Vannevar Bush 的 Memex（1945）相关——一个个人的、策划的知识库，文档之间有关联轨迹。Bush 的愿景比网络变成的样子更接近这个：私人的、积极策划的、文档之间的连接与文档本身一样有价值。他无法解决的部分是谁来做维护。LLM 处理这个。

## 注意

本文档故意保持抽象。它描述的是想法，而不是具体的实现。确切的目录结构、schema 约定、页面格式、工具——所有这些都将取决于你的领域、你的偏好和你选择的 LLM。上面提到的一切都是可选的和模块化的——选择有用的，忽略没用的。例如：你的来源可能只是文本，所以你根本不需要图像处理。你的 wiki 可能足够小，索引文件就是你所需要的，不需要搜索引擎。你可能不关心幻灯片，只想要 markdown 页面。你可能想要一套完全不同的输出格式。使用这个的正确方法是与你的 LLM 代理分享它，并共同努力创建一个适合你需求的版本。本文档的唯一工作是传达这个模式。你的 LLM 可以弄清楚其余的。
